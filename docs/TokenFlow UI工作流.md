# TokenFlow UI工作流

## 1. 目的

这份文档不是新的视觉方案，而是把 TokenFlow 当前项目适合采用的 UI 工作方式固定下来。

适用范围：

- `React 19 + Vite + Tauri 2`
- `Tailwind CSS 4`
- `shadcn/ui + Radix`
- 桌面端优先的管理台 / 监控面板场景

当前代码基线可参考：

- `src/App.tsx`
- `src/index.css`
- `src/components/Dashboard.tsx`
- `src/components/Sidebar.tsx`
- `src/components/AddAccountDialog.tsx`
- `src/components/SettingsPage.tsx`

## 2. 当前项目的 UI 方法论

TokenFlow 更适合走“组件驱动设计”，不适合走“先大面积出设计稿，再回头切页面”的路线。

原因：

- 现有页面已经建立了稳定的 `shadcn` 组件基底和 token 变量体系。
- 这是一个 Tauri 桌面应用，交互密度、状态复杂度、空态和异常态比纯营销页更重要。
- 页面主体是 dashboard、provider 监控、日志、设置，核心价值来自信息层级和操作效率，不是复杂视觉插画。

因此后续 UI 需求默认按下面顺序推进：

1. 先整理信息结构和状态分层。
2. 再决定复用哪些现有组件。
3. 最后才做视觉强化和细节 polish。

## 3. 新装技能的使用时机

本地已安装：

- `web-design-guidelines`
- `web-accessibility`

建议用法：

### 3.1 `web-design-guidelines`

适合在这些场景触发：

- 新页面或新模块从 0 到 1 设计
- 旧页面需要重排信息层级
- dashboard 卡片、列表、过滤器、工具栏需要统一风格
- 需要决定“先放什么、弱化什么、按钮应该放哪里”

在 TokenFlow 里，优先用于：

- `Dashboard`
- `Providers`
- `Logs`
- `Settings`
- 新增的 provider 接入页面

建议提示词风格：

```text
使用 web-design-guidelines，基于 TokenFlow 当前的 shadcn/ui + Tailwind 4 + Tauri 桌面应用风格，重构 [页面/组件名] 的信息层级、布局分区、操作优先级和空态，不改变现有数据模型。
```

### 3.2 `web-accessibility`

适合在这些场景触发：

- 新增 Dialog、Form、Popover、Tabs、筛选器
- 修改键盘交互
- 调整颜色、状态提示、错误提示
- 页面改版后做收尾验收

在 TokenFlow 里，优先用于：

- `AddAccountDialog`
- `ApiKeyDialog`
- `DeviceFlowDialog`
- `OAuthFlowDialog`
- `SettingsPage`
- `LogsPage` 的筛选与搜索

建议提示词风格：

```text
使用 web-accessibility，检查 TokenFlow 当前 [页面/组件名] 的键盘可达性、焦点顺序、语义标签、错误提示、颜色对比和屏幕阅读器友好性，并直接给出修复建议或补丁。
```

## 4. TokenFlow 的标准 UI 交付流程

### 4.1 第一步：先定义任务类型

开始前先把需求归到这 4 类之一：

- 布局类：版块重排、卡片分组、工具栏调整
- 组件类：新增卡片、对话框、表单、筛选器、图表区块
- 视觉类：颜色、层次、密度、间距、品牌感
- 质量类：无障碍、空态、错误态、加载态、响应式

如果一个需求同时包含多类，默认顺序是：

1. 布局
2. 组件
3. 视觉
4. 质量

### 4.2 第二步：先查复用点，再动代码

任何 UI 变更先检查这 4 层：

1. `src/components/ui/*`
2. 同页已有模式
3. 相邻页面已有模式
4. `src/index.css` 里的主题 token

优先原则：

- 优先复用现有 `Button`、`Card`、`Dialog`、`Badge`。
- 优先沿用当前圆角、边框透明度、卡片层次。
- 优先改组合方式，不轻易新造基础组件。

只有在以下情况才新增基础组件：

- 同一模式已经出现至少 3 次
- 现有组件组合明显重复
- 抽出来后会降低页面复杂度

### 4.3 第三步：先补状态，再补样式

每个 UI 需求都必须同时考虑这几种状态：

- 默认态
- loading
- empty
- error
- success 或 completed
- disabled
- 隐私模式下的展示差异

在 TokenFlow 里，很多页面已经有这个趋势，但后续要固定成规则，而不是想到哪里补到哪里。

### 4.4 第四步：组件实现顺序

推荐顺序：

1. 先写静态结构
2. 接入真实数据
3. 补齐 loading / empty / error
4. 接入交互
5. 最后调视觉细节

不建议一开始就先调颜色和动画。

## 5. 项目内的页面级规则

### 5.1 Dashboard

目标是“一眼看清当前状态”，不是展示所有信息。

规则：

- 第一屏优先放总览指标和风险信号
- 第二层放趋势和分布
- 第三层放 provider / account 细节
- 不要让所有卡片权重一样

适合做强弱分层：

- 强：核心指标、风险、剩余额度
- 中：趋势、分布、最近活动
- 弱：说明文字、辅助标签、背景装饰

### 5.2 Providers

目标是“深入单一 provider 并快速操作”。

规则：

- 顶部 tab 只负责 provider 切换
- 页头统计只保留该 provider 最重要的 3 个数字
- 账号卡片里把“状态、额度、操作”拆成稳定分区
- 高频操作固定在卡片上半区，不要埋到说明文字附近

### 5.3 Logs

目标是“快速筛选和定位问题”。

规则：

- 筛选器和搜索必须固定在视觉上方
- 表格列优先保证可扫描性，不追求塞更多字段
- 状态码、耗时、provider 这些信号要比描述文字更显眼
- 空日志和加载中的界面不能长得太像

### 5.4 Settings

目标是“改配置时不出错”。

规则：

- 设置项按风险和频率分组
- 危险操作与普通设置拉开距离
- Toggle 旁边尽量给一句后果说明
- 颜色配置、隐私配置、启动配置不要混排在同一视觉块里

## 6. 视觉与代码约束

### 6.1 视觉约束

- 保持当前浅色主题为默认主线，深色主题作为完整映射而不是临时反色
- 延续现有圆角体系：`rounded-2xl` 到 `rounded-[28px]`
- 延续现有“轻边框 + 半透明卡片 + 柔和背景”的桌面控制台风格
- provider 色彩只承担“识别”作用，不承担大面积背景职责

### 6.2 代码约束

- 主题变量统一放在 `src/index.css`
- 组件变体优先放在 `src/components/ui/*`
- 页面编排放在页面组件，不把页面布局逻辑塞进基础组件
- 一次需求如果只是重排布局，尽量不要同时重写 copy 和数据逻辑

## 7. 无障碍验收清单

每次 UI 改动完成后，至少做下面检查：

1. 所有可交互元素都能用键盘访问
2. 焦点顺序符合视觉顺序
3. Dialog 打开后焦点进入内容区，关闭后返回触发点
4. 输入框有可见标签或可理解名称
5. 错误提示不是只靠颜色表达
6. 危险按钮与普通按钮有足够区分
7. 占位文字不是唯一说明来源
8. 深浅主题下文字和边框对比足够

优先检查组件：

- 表单
- Dialog
- Tabs
- 自定义 Toggle
- 颜色选择器
- 搜索和筛选区

## 8. 一次完整需求的推荐执行模板

以“新增一个 provider 详情面板”为例：

1. 先说明这个面板的主任务是什么。
2. 列出必须展示的 3 到 5 个关键信号。
3. 参考现有 `Card` 和 `Button` 组合出静态骨架。
4. 接入真实 provider 数据。
5. 补 `loading / empty / error / no quota windows`。
6. 用 `web-design-guidelines` 检查层级和布局密度。
7. 用 `web-accessibility` 检查键盘、标签、对比度和反馈文案。
8. 最后再决定是否需要更强的视觉强调。

## 9. 团队内的默认约定

后续做 TokenFlow 的 UI 需求，默认遵循下面约定：

- 不为了“看起来更设计”而偏离现有 `shadcn` 体系
- 不轻易引入新的大 UI 库
- 不先画复杂稿再找代码落地
- 先保证监控信息好读、操作路径短、状态可解释
- 每次改 UI 都把无障碍检查作为交付的一部分

## 10. 推荐的日常协作方式

如果是常规 UI 需求，建议按这个节奏：

1. 先描述目标页面和核心任务
2. 让我用 `web-design-guidelines` 做结构方案
3. 直接在现有组件体系里实现
4. 再用 `web-accessibility` 做收尾检查
5. 最后做一次本地交互验证

这条流程比“先做一版视觉稿，再手工翻译成代码”更适合当前 TokenFlow。
