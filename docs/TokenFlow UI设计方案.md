# TokenFlow UI设计方案

## 1. 设计目标

TokenFlow 的 UI 首要目标不是“信息堆得多”，而是让用户在最短时间内看清四件事：

1. 现在一共连了多少个平台、多少账号。
2. 哪些平台和账号的额度最危险。
3. 哪些平台最近消耗最多 token、请求最多。
4. 某个平台内部每个账号具体是什么状态。

因此 UI 必须围绕以下原则展开：

- 固定左侧导航，右侧内容区承担全部工作流。
- Dashboard 看全局，Providers 看平台，Logs 看过程，Settings 管行为，About 建立信任。
- 账号结构统一，额度语义保持平台原生，不再为了统一而统一。
- 没有真实数据的地方明确空态，不使用假数据伪装成熟。

## 2. 总体布局

### 2.1 页面结构

应用整体采用：

- 左侧固定导航栏
- 右侧内容区
- 内容区内部独立滚动

左栏固定 5 个入口：

1. Dashboard
2. Providers
3. Logs
4. Settings
5. About

### 2.2 导航原则

- 左栏只承载一级导航，不再混入 provider 列表。
- Provider 切换放在 Providers 页顶部，用标签页风格承载。
- 左栏视觉应稳定，避免随账号数量变化而抖动。

## 3. Dashboard 设计

Dashboard 是默认首页，整页可上下滚动，分上中下三段。

### 3.1 顶部总览区

顶部显示 4 个核心指标卡：

- 总账号数量
- Token 总消耗量
- 请求总数量
- Provider 连接数量

#### 统计口径

- 默认统计周期为近 30 天。
- 总账号数量按已连接账号计算。
- Provider 连接数量按至少存在 1 个已连接账号的 provider 计算。
- Token 总消耗量和请求数来自请求日志层。
- 若请求日志层未接入，则显示 0，并提示“等待请求追踪数据”。

### 3.2 中部月度柱状图

图表目标：

- 显示近 30 天每天的 token 消耗量。
- 每一天是一根堆叠柱。
- 每种颜色对应一个 provider。
- provider 颜色允许在 Settings 页自定义。

#### 图表结构

- X 轴：近 30 天日期
- Y 轴：每日 token 消耗量
- 图例：provider 颜色映射
- hover：显示当天总 token 与各 provider 贡献

#### 空态策略

如果请求日志尚未接入：

- 图表容器保留
- 中央显示空态说明
- 不显示伪造柱子

### 3.3 底部供应商与账号总览

这是 Dashboard 最重要的区域。

#### 展示规则

- 只显示存在已连接账号的 provider。
- 每个 provider 一个大圆角矩形卡片。
- provider 卡片内部包含多个账号小卡片。
- 尽量控制高度，保证一屏能看到尽量多的平台。

#### Provider 大卡片内容

- Provider Logo
- Provider 名称
- 已连接账号数量
- 近 30 天 token 消耗
- 近 30 天请求数

#### 账号小卡片内容

- 左侧：邮箱或用户自定义名称
- 右侧：平台原生额度圆环
- 下方：近 30 天 token 消耗 / 请求数 / 最近同步时间

#### 原生额度展示

- OpenAI：显示 5h + Weekly 等多个圆环
- Copilot：显示 Monthly 单圆环
- Cursor / Trae：显示其原生可读窗口
- Vertex / Qwen / iFlow 这类暂时无窗口的平台显示“已连接但暂无额度窗口”

#### 交互

- 点击 provider 卡片可跳到 Providers 页并定位到对应 provider tab。
- 点击账号小卡片后续可以扩展为打开账号详情抽屉。

## 4. Providers 页设计

Providers 页用于深入查看某一个平台。

### 4.1 顶部 provider tabs

Provider 切换采用浏览器标签页风格：

- 每个 provider 一个 tab
- tab 显示 provider 名称与账号数量
- 当前 tab 高亮

### 4.2 页头区域

页头包含：

- 左侧：Provider Logo + 名称
- 右侧：添加账号按钮

中间显示 3 个聚合统计：

- 账号数量
- Token 消耗数
- 请求数

默认口径为近 30 天。

### 4.3 账号详情区

每个账号一个独立矩形面板。

账号面板包含：

- 邮箱
- 用户名或自定义名称
- 平台原生额度圆环组
- Token 消耗数
- 请求数
- 近 30 天 token 消耗趋势曲线
- 最近同步时间

#### 设计原则

- 不同 provider 的额度窗口数量可以不同。
- 账号面板布局统一，额度语义不统一。
- 趋势曲线和请求统计来自请求日志层。
- 若当前账号尚无请求活动，则趋势区域显示空态。

## 5. Logs 页设计

Logs 页承担两个目的：

1. 看请求过程
2. 看应用调试与诊断日志

### 5.1 顶部 tab

日志页顶部使用二级 tab：

- Requests
- App Logs

### 5.2 Requests 视图

顶部摘要区显示：

- 总请求数
- 成功率
- Token 数
- 平均耗时

右侧过滤区包含：

- Provider 筛选
- 状态码筛选
- 搜索框（模型 / provider）

#### 请求表格字段

- Time
- Provider
- Model
- Status
- Input / Output Tokens
- Duration

#### 视觉规则

- 200 使用绿色状态胶囊
- 4xx 使用橙色
- 5xx 使用红色
- provider 名称可配合彩色标记

### 5.3 App Logs 视图

App Logs 用于承接当前已有的调试日志能力。

包含：

- 刷新日志
- 复制日志
- 清空日志
- 日志文件路径
- 结构化日志列表

每条日志包含：

- 时间
- scope
- message

## 6. Settings 页设计

Settings 页聚焦行为设置，不承载调试日志。

### 6.1 语言

- 中文
- English

### 6.2 主题

- 浅色
- 深色

### 6.3 隐私模式

开启后：

- 邮箱脱敏
- 用户名脱敏
- Dashboard、Providers、Logs 全局生效

#### 脱敏规则

- 邮箱：`a***z@domain.com`
- 用户名：保留首尾，中间替换为 `***`

### 6.4 启动行为

- 关闭窗口时最小化到托盘
- 开机自启动

说明：

- 最小化到托盘应尽量实际生效。
- 开机自启动若系统集成尚未完成，可先保留偏好设置与说明。

### 6.5 Provider 颜色

用于自定义：

- Dashboard 柱状图颜色
- Logs 页 provider 标记颜色
- 未来可扩展到 provider badge 与 legend

## 7. About 页设计

About 页负责产品信任感。

包含：

- 产品介绍
- 当前版本
- 检查更新
- 项目地址
- 捐赠入口

后续更新检查、发布说明等也应汇聚在这里。

## 8. 数据口径与空态策略

### 8.1 当前真实可用数据

当前版本可稳定提供：

- 账号列表
- provider 连接状态
- provider 原生额度窗口
- 最近同步时间
- 本地 debug logs

### 8.2 当前缺失但 UI 需要预留的数据

以下数据是本次 UI 需要预留位置，但当前可能为空：

- 请求日志明细
- 近 30 天 token 消耗
- 近 30 天请求数
- 请求成功率
- 平均耗时

### 8.3 空态原则

缺失数据时必须：

- 展示容器结构
- 明确说明“等待请求追踪数据”
- 不展示假数字、假曲线、假表格

## 9. 视觉语言建议

- 左栏固定宽度，强调稳定
- 主内容区大量使用圆角卡片，但不过度堆叠
- Provider 颜色系统贯穿 Dashboard / Providers / Logs
- 浅色主题优先，深色主题作为完整切换方案存在
- Dashboard 偏总览
- Providers 偏分析
- Logs 偏工具化

## 10. 首轮重构落地范围

首轮 UI 重构优先完成：

1. 左栏五页结构
2. Dashboard 三段式布局
3. Providers 顶部 tab 化
4. Logs 双 tab 化
5. Settings 重构为行为设置
6. About 独立页面

首轮允许保留空态的数据位：

- 请求日志统计
- 月度 token 柱状图真实堆叠数据
- 账号级 30 天 token 趋势真实曲线

## 11. 后续 UI 迭代

第二轮可继续补：

- 请求日志真实接入
- Dashboard 图例显隐
- 账号详情抽屉
- Provider 页排序与筛选
- Logs 导出能力
- About 页更新检测
