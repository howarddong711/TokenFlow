# TokenFlow 竞品研究与设计文档

## 1. 文档目的
这份文档用于回答 4 个问题：

1. 现在市场上“适合做额度/余额监控”的 AI Coding Plan 平台有哪些
2. 这些平台各自有哪些会员计划 / 订阅档位
3. `CLIProxyAPI`、`Quotio`、`Win-CodexBar` 这三个项目分别是怎么实现额度监控的
4. 和 TokenFlow 当前初版目标相比，这些项目已经做了哪些额外能力

这份文档的定位不是营销分析，而是给 TokenFlow 后续产品设计和技术架构直接提供输入。

## 2. 先给结论
### 2.1 TokenFlow 的真正竞品分成两类
- 第一类是“额度聚合器 / 状态栏监控器”，最接近的是 `Quotio` 和 `Win-CodexBar`
- 第二类是“认证与代理底座”，最接近的是 `CLIProxyAPI`

对 TokenFlow 来说，真正值得学习的不是某一个 UI，而是这三个项目组合起来形成的能力链路：

- `CLIProxyAPI` 解决“怎么登录、怎么持久化、怎么多账号、怎么做统一代理”
- `Quotio` 解决“怎么把多个 provider 的额度抓回来，并放进一个桌面产品里”
- `Win-CodexBar` 解决“怎么把额度监控抽象成统一 provider 框架，并支持很多来源和很多平台”

### 2.2 TokenFlow 初版最应该优先覆盖的平台
按“用户需求强 + 额度结构明确 + 技术路径相对可行”排序，建议：

- P0：ChatGPT / Codex、Claude、GitHub Copilot、Cursor
- P1：Kiro、Warp、GLM Coding Plan、Trae
- P2：Gemini、Windsurf / Factory、OpenCode、Antigravity、JetBrains AI、OpenRouter

其中最重要的现实判断是：

- `Codex`、`Claude`、`Copilot`、`Cursor` 都已经在参考项目里证明“可监控”
- `Gemini CLI` 在 `Quotio` 里目前只能稳定拿到账户信息，拿不到真正公开额度
- 有些平台不是“订阅配额”，而是“API credits”或“workspace quota”，TokenFlow 数据模型必须允许它们共存

### 2.3 如果 TokenFlow 只想做对初版，最值得抄的不是“更多平台”，而是 3 个架构决定
- Provider Adapter：每个平台独立实现，不共享抓取细节
- Source Strategy：同一平台允许多个数据源和回退链路
- Normalized Quota Model：统一存“计划、窗口、剩余、重置、状态”，不要把不同平台硬塞成一个数字

## 3. 市场平台地图
下面这张表只统计“对 TokenFlow 有意义”的平台。这里的“有意义”指至少满足以下之一：

- 有明确的订阅计划 / 使用额度
- 有用户会关心是否还剩可用配额
- 在参考项目里已经出现，说明真实需求存在

### 3.1 第一层：应重点跟踪的主流平台
| 平台 | 平台类型 | 当前可识别的计划 / 订阅 | 额度形态 | 备注 |
|---|---|---|---|---|
| ChatGPT / Codex | 官方 AI Coding / ChatGPT 体系 | Free、Plus、Pro、Business、Enterprise | 会话窗口、周窗口、Credits、模型权限 | TokenFlow 必做 |
| Claude / Claude Code | 官方 AI Coding 体系 | Free、Pro、Max、Team、Enterprise | 5h、7d、模型级窗口、额外 credits | TokenFlow 必做 |
| GitHub Copilot | 官方 IDE / Coding 订阅 | Free、Pro、Pro+、Business、Enterprise | 月度 interactions / completions / premium usage | TokenFlow 必做 |
| Cursor | AI IDE | Free、Pro、Business、Ultra | 月度计划额度 + on-demand | TokenFlow 必做 |
| Gemini | Google 消费级 + 企业级双体系 | Google AI Free / Pro / Ultra；Code Assist Standard / Enterprise | quota、模型权限、workspace / cloud 配额 | 需要拆成两个产品族 |
| Kiro | AWS / IDE 编程产品 | 代码中可见 subscription + usage limits | 月度 credits / usage breakdown | 技术可行，价值较高 |
| Windsurf / Factory / Droid | AI IDE / Codeium 体系 | 公开计划存在，但额度结构较碎 | 订阅额度 + workspace / model 使用 | 适合 P1/P2 |

### 3.2 第二层：参考项目中已支持、值得纳入候选池的平台
| 平台 | 参考项目出现情况 | 可监控对象 | 监控成熟度 |
|---|---|---|---|
| Warp | `Quotio`、`Win-CodexBar` | request credits、bonus credits | 高 |
| GLM / z.ai Coding Plan | `CLIProxyAPI`、`Quotio`、`Win-CodexBar` | token quota、MCP quota | 高 |
| Trae | `Quotio` | entitlement + 多类请求额度 | 中高 |
| Antigravity | `CLIProxyAPI`、`Quotio`、`Win-CodexBar` | quota fraction、reset time、订阅 tier | 中高 |
| OpenCode | `CLIProxyAPI`、`Win-CodexBar` | rolling / weekly usage | 中 |
| JetBrains AI | `Win-CodexBar` | AI quota / token / account | 中 |
| OpenRouter | `Win-CodexBar` | credit balance、key quota | 高，但属于 API credits，不是会员 plan |
| Augment | `Win-CodexBar` | credits | 中 |
| MiniMax | `Win-CodexBar` | coding plan remains | 中 |
| Kimi / Kimi K2 | `CLIProxyAPI`、`Win-CodexBar` | 5h / weekly / credits | 中 |
| Qwen Code | `CLIProxyAPI` | 账号与调用路由 | 中，余额监控侧证据偏少 |
| iFlow | `CLIProxyAPI`、`Quotio` | OAuth 与代理接入 | 中，余额监控侧证据偏少 |
| Vertex AI | `CLIProxyAPI`、`Quotio`、`Win-CodexBar` | cloud quota / cost | 更接近云配额，不是普通订阅会员 |

### 3.3 第三层：对 TokenFlow 要特别注意的边界
这些对象不能简单和“会员余额”画等号：

- OpenRouter：更像 API credit 余额
- Vertex AI：更像 GCP 项目配额 / 账单
- Warp：既有订阅请求额度，也有 bonus credits
- GLM / z.ai：更像 coding plan + token / MCP 双配额
- Gemini：消费者计划与企业 Code Assist 是两个体系

结论是：TokenFlow 的数据模型不能只支持一个 `remainingCount`。

## 4. 当前平台订阅计划整理
这一节只做产品设计需要的“计划层级识别”，不追求把所有价格写成报价单。

### 4.1 官方公开、且对 TokenFlow 最关键的平台
#### ChatGPT / Codex
- 官方公开主线可识别计划：Free、Plus、Pro、Business、Enterprise
- 在参考代码里还能识别到 `team` / `plan_type`
- 对 TokenFlow 的关键点：
  - 计划名可以从 JWT / usage API 中提取
  - 额度不是单一月包，更像会话窗口 + 周窗口 + credits 组合

#### Claude / Claude Code
- 官方公开主线可识别计划：Free、Pro、Max、Team、Enterprise
- 在参考代码里，Claude Code 监控重点不是“一个总量”，而是多个窗口：
  - 5 小时窗口
  - 7 天窗口
  - Sonnet / Opus 等模型级窗口
  - extra usage / monthly credits

#### GitHub Copilot
- 官方公开主线可识别计划：Free、Pro、Pro+、Business、Enterprise
- 在 `Quotio` 代码中还能进一步区分：
  - free limited
  - individual / pro
  - business
  - enterprise
- 对 TokenFlow 的关键点：
  - Copilot 是典型“月配额 + 不同能力项”的产品
  - 不应只展示一个总余额，至少要支持 chat / completions / premium interactions 多项

#### Cursor
- 官方公开主线可识别计划：Free、Pro、Business、Ultra
- 在参考代码里可观测到的额度形态：
  - plan usage
  - on-demand usage
  - billing cycle
- 对 TokenFlow 的关键点：
  - Cursor 不是简单“剩余次数”
  - 既有计划内额度，也可能有额外按需消费

### 4.2 官方或代码中可识别，但更适合 P1/P2 的平台
#### Gemini
- 需要拆成两个产品族看：
  - Google AI 消费者计划：Free / Pro / Ultra
  - Google Code Assist / 企业体系：Standard / Enterprise 等
- 现实问题：
  - `Quotio` 目前只能稳定识别 Gemini CLI 账号，不具备公开可用的真实额度 API
  - 所以 Gemini 对 TokenFlow 来说更适合先做“账号状态 + 计划占位”，再看能否补额度

#### Kiro
- `Quotio` 代码能识别 subscriptionInfo、usageBreakdown、nextDateReset
- 更像“多资源项 + 月度重置”的额度模式

#### Trae
- `Quotio` 代码直接识别计划：
  - Free
  - Pro
  - Team
  - Builder
- 同时识别多类额度：
  - advanced model
  - auto completion
  - premium fast
  - premium slow

#### Warp
- 不是传统“会员剩余次数”，而是：
  - request limit
  - bonus grants / bonus credits
- 适合 TokenFlow 作为“credits 型 provider”支持

#### GLM / z.ai Coding Plan
- 监控对象更接近：
  - token quota
  - MCP quota
  - reset time
- 对 TokenFlow 很有价值，因为这类平台很适合“统一配额聚合”场景

## 5. 三个项目的代码研究
## 5.1 CLIProxyAPI
### 项目定位
`CLIProxyAPI` 本质上不是“额度可视化应用”，而是“多 provider OAuth / API key / 多账号代理底座”。

它最核心的价值不是展示余额，而是：

- 统一登录
- 统一认证持久化
- 统一模型路由
- 多账号轮询与降级
- 管理 API

### 支持的平台
从 README 和代码可确认的主平台包括：

- Codex / OpenAI
- Claude
- Gemini
- Qwen
- iFlow
- Antigravity
- GitHub Copilot
- Kimi
- Vertex / OpenAI-compatible provider

### 它如何处理“额度 / 余额相关信息”
CLIProxyAPI 的“额度相关”实现，更多是给上层产品提供底层能力，而不是自己完成完整额度面板：

#### 1. 认证文件与计划信息提取
- Codex 登录后会从 `id_token` 里提取：
  - `chatgpt_account_id`
  - `chatgpt_plan_type`
  - `chatgpt_subscription_active_until`
- 这说明 CLIProxyAPI 已经具备“识别计划类型”的底层能力

#### 2. Quota Exhausted / Cooling 机制
- `sdk/cliproxy/auth/types.go` 和 `conductor.go` 维护了 `QuotaState`
- 当账号遇到 quota exceeded 时，会进入冷却、记录 `NextRecoverAt`、回退等级、不可用状态
- 这更像“运行态调度配额”，不是“用户可视化余额”

#### 3. 使用量统计
- 管理 API 提供 `/usage`
- 统计的是请求数、失败数、token usage、provider usage record
- 这是“代理请求统计”，不是“官方订阅剩余额度”

#### 4. provider 侧的 usage 元数据解析
- 在 runtime executor 中，会从不同 provider 的响应里提取 usage
- 例如 OpenAI / Claude / Gemini / Antigravity / Qwen 等都有自己的 usage parser
- 这更像“请求消耗统计”，而不是“官方账号余额面板”

### 对 TokenFlow 的启发
CLIProxyAPI 最值得 TokenFlow 借鉴的不是 UI，而是这 4 个基础层：

- 统一 auth record 模型
- 多 provider executor 注册机制
- quota exceeded 冷却与可恢复状态
- 管理 API / 本地后台服务

### 相比 TokenFlow 的额外能力
相比你的初版目标，CLIProxyAPI 额外做了很多“平台基础设施”能力：

- 多账号负载均衡
- 模型路由与 fallback
- OpenAI / Claude / Gemini 兼容 API
- 本地代理服务
- SDK 可嵌入
- watcher / 热重载
- 请求日志与使用统计
- CLI / TUI / 管理 API

结论：CLIProxyAPI 不是 TokenFlow 的直接产品形态竞品，但很像一个可借鉴的“后端内核”。

## 5.2 Quotio
### 项目定位
`Quotio` 是目前最接近 TokenFlow 目标的参考项目之一。

它的定位是：

- macOS 原生桌面应用
- 围绕 CLIProxyAPI 构建
- 同时支持 quota-only 模式与 full proxy 模式
- 把多平台账号、额度、代理、配置统一到一个产品里

### 支持的平台
从 `AIProvider`、README 和各个 `QuotaFetcher` 可确认的平台包括：

- Gemini
- Claude
- Codex / OpenAI
- Qwen
- iFlow
- Antigravity
- Vertex
- Kiro
- GitHub Copilot
- Cursor
- Trae
- GLM
- Warp

### 它实现余额监控的方式
Quotio 的关键价值在于：每个平台都可以用不同的抓取方式，但最后统一归一成 `ProviderQuotaData`。

#### 1. Codex / OpenAI
- 数据源：
  - `~/.cli-proxy-api` 的 auth 文件
  - quota-only 模式下也支持直接读 `~/.codex/auth.json`
- 方式：
  - 解析 JWT 拿 email / `plan_type`
  - 过期时 refresh token
  - 调用 `https://chatgpt.com/backend-api/wham/usage`
- 能拿到：
  - 计划类型
  - session window
  - weekly window
  - reset time
  - credits

#### 2. Claude
- 数据源：
  - 本地 auth 文件
- 方式：
  - 读取 access token / refresh token
  - 过期自动刷新
  - 调用 `https://api.anthropic.com/api/oauth/usage`
- 能拿到：
  - 5h
  - 7d
  - Sonnet / Opus usage
  - extra usage

#### 3. GitHub Copilot
- 数据源：
  - `~/.cli-proxy-api` 下的 `github-copilot-*.json`
- 方式：
  - 调 GitHub Copilot entitlement API
- 能拿到：
  - plan 类型
  - monthly quotas
  - quota snapshots
  - reset date

#### 4. Cursor
- 数据源：
  - Cursor 的 `state.vscdb`
- 方式：
  - 先从 SQLite 读取 access token、membershipType、subscriptionStatus
  - 再调 Cursor usage-summary API
- 能拿到：
  - 会员类型
  - billing cycle
  - plan usage
  - on-demand usage

#### 5. Trae
- 数据源：
  - Trae 的 `storage.json`
- 方式：
  - 读取 JWT / account 信息
  - 调 Trae entitlement API
- 能拿到：
  - plan 类型
  - advanced / auto-completion / premium fast / premium slow 等多项额度
  - reset time

#### 6. Kiro
- 数据源：
  - 本地 AWS / Kiro token 文件
- 方式：
  - 读 token
  - 到期前主动 refresh
  - 401/403 后被动 refresh
  - 调 `getUsageLimits`
- 能拿到：
  - subscriptionInfo
  - usageBreakdown
  - reset time

#### 7. Antigravity
- 数据源：
  - 本地语言服务 / 数据库 / token
- 方式：
  - 抓 quota
  - 额外缓存 subscription 信息，避免重复 API 调用
- 能拿到：
  - model quota
  - reset time
  - subscription tier

#### 8. GLM
- 数据源：
  - API Key
- 方式：
  - 调 `https://bigmodel.cn/api/monitor/usage/quota/limit`
- 能拿到：
  - token quota
  - MCP quota
  - remaining
  - next reset time

#### 9. Warp
- 数据源：
  - API Key
- 方式：
  - 调 Warp API / GraphQL
- 能拿到：
  - request credits
  - bonus credits
  - next refresh

#### 10. Gemini CLI
- 数据源：
  - `~/.gemini/oauth_creds.json`
  - `~/.gemini/google_accounts.json`
- 方式：
  - 只读 auth file / JWT
- 限制：
  - 当前没有公开、稳定的 quota API
  - 只能作为“账号已连接 + 计划未知 / 配额未知”的占位

### Quotio 的重要产品能力
Quotio 比 TokenFlow 当前目标多做了很多东西：

- quota-only 模式，不依赖 proxy 就能看额度
- full mode，直接管理 CLIProxyAPI
- menu bar 常驻入口
- agent auto configuration
- logs / dashboard / API key 管理
- fallback routing / round robin / fill first
- 通知、自动刷新、低额度提醒
- Cloudflare tunnel / remote mode
- warmup 模型预热
- 多语言

### 对 TokenFlow 的启发
Quotio 证明了一件非常重要的事：

**真正可用的 quota aggregator，不会只有一种抓取方式。**

它必须同时支持：

- OAuth API
- 本地 auth 文件
- 本地 SQLite / JSON
- 浏览器态 / cookies
- API key
- 代理服务管理

## 5.3 Win-CodexBar
### 项目定位
`Win-CodexBar` 是 Windows 原生的额度监控器，定位上与 TokenFlow 也非常接近。

它更像：

- 多 provider usage monitor
- 原生系统托盘应用
- provider framework + fetch strategy 框架

### 支持的平台
从 `ProviderId` 可确认支持 21 个 provider：

- Codex
- Claude
- Cursor
- Factory
- Gemini
- Antigravity
- Copilot
- z.ai
- MiniMax
- Kiro
- Vertex AI
- Augment
- OpenCode
- Kimi
- Kimi K2
- Amp
- Warp
- Ollama
- OpenRouter
- Synthetic
- JetBrains AI

### 它实现余额监控的方式
Win-CodexBar 的最大特点是把“数据源策略”做成了标准能力。

它定义了统一的：

- `Provider`
- `ProviderMetadata`
- `SourceMode`（Auto / OAuth / Web / Cli）
- `UsageSnapshot`
- `RateWindow`

也就是说，它不是每个平台随便写一个抓取脚本，而是把平台监控抽象成统一框架。

#### 典型抓取策略
#### Codex
- OAuth API
- CLI RPC / CLI PTY fallback
- 还支持本地 JSONL usage / cost 扫描

#### Claude
- Auto 链路：OAuth -> Web cookies -> CLI
- 是目前最完整的多源回退实现之一

#### Cursor
- Web API + browser cookies

#### Copilot
- Device Flow / API token

#### OpenCode
- browser cookies
- 调内部 server endpoint，解析 rolling / weekly usage

#### Warp
- API key
- GraphQL 查询 request limit 和 bonus credits

#### z.ai / GLM
- API token
- quota API

#### Kiro
- CLI / token / usage endpoint

#### Factory / Windsurf
- cookies、stored tokens、local storage、WorkOS cookies 多重回退

### Win-CodexBar 的额外能力
相比 TokenFlow 当前目标，Win-CodexBar 还额外做了：

- 系统托盘双条进度图标
- CLI 命令行输出：`usage` / `cost`
- 浏览器 cookie 自动提取
- Windows Credential Manager 集成
- token / cost / daily usage 数据模型
- session pace / reset 倒计时
- provider 状态提醒与通知
- status page / dashboard URL
- 本地日志扫描估算成本

### 对 TokenFlow 的启发
Win-CodexBar 对 TokenFlow 最有价值的不是“又支持了几个 provider”，而是它的抽象方式：

- 平台是插件化的
- 数据来源是策略化的
- 输出模型是统一的
- UI 层不关心平台细节，只消费标准快照

这非常适合 TokenFlow 后续架构。

## 6. 对比 TokenFlow 当前目标，这些项目多做了什么
你的当前目标核心是：

- 各种 coding plan 的余额统计
- 标准登录
- 浏览器回调
- 长期登录态
- 多账号

而竞品们在此基础上已经多做了下面这些东西。

### 6.1 代理与路由能力
- 本地代理
- provider 兼容 API
- 模型映射
- fallback routing
- round robin / fill first

这类能力以 `CLIProxyAPI`、`Quotio` 为主。

### 6.2 系统级入口
- 菜单栏 / 托盘常驻
- 动态图标
- 快捷入口
- 通知提醒

这类能力以 `Quotio`、`Win-CodexBar` 为主。

### 6.3 自动配置开发工具
- 自动检测 CLI agent
- 自动写 shell 配置
- 一键配置 Claude Code / Codex CLI / Gemini CLI / OpenCode / Amp 等

这类能力以 `Quotio` 为主。

### 6.4 使用分析，而不只是余额展示
- 请求流量
- token usage
- 成本估算
- daily usage
- 失败率
- pace 预测

这类能力以 `CLIProxyAPI`、`Win-CodexBar` 为主。

### 6.5 多源抓取与隐式账号发现
- 浏览器 cookie
- SQLite
- 本地配置文件
- CLI 输出
- OAuth API
- 本地 service / language server

这类能力以 `Quotio`、`Win-CodexBar` 最成熟。

### 6.6 远程连接与运维能力
- remote management
- tunnel
- updater
- health / logs / config editing

这类能力不适合 TokenFlow 初版，但说明产品未来上限很高。

## 7. 对 TokenFlow 的设计输入
## 7.1 TokenFlow 初版不该怎么做
### 不要把所有平台都设计成一个字段
不要只做：

- 平台名
- 会员名
- 剩余百分比

因为真实平台里至少已经出现了这些不同结构：

- 3 小时窗口 + 周窗口
- 5 小时窗口 + 7 天窗口
- model-specific window
- 月度 credits
- request credits + bonus credits
- token quota + MCP quota
- entitlement pack 里的多个子额度

### 不要把“登录”和“监控”耦死
有些平台靠 OAuth API，有些靠 cookie，有些靠本地文件，有些靠 API key。

所以 TokenFlow 应该拆开：

- 认证接入方式
- 配额抓取方式
- 展示方式

## 7.3 TokenFlow 的 provider 设计建议
建议采用下面的结构：

### ProviderAdapter
职责：

- 定义这个平台支持哪些 source
- 定义账号发现逻辑
- 定义 quota fetch 逻辑
- 定义 plan normalize 逻辑

### FetchStrategy
职责：

- 真正执行一次抓取
- 输出统一的 `AccountQuotaSnapshot`

### MergePolicy
职责：

- 同平台多个来源时如何合并
- 例如 OAuth API 优先于本地文件
- 或者“计划名来自 JWT，额度来自 usage API”

这套思路本质上是把 `Quotio` 的 fetcher 思路和 `Win-CodexBar` 的 provider/source 抽象结合起来。

## 7.4 TokenFlow 的推荐平台优先级
### P0：首批必须打磨到可用
- ChatGPT / Codex
- Claude
- GitHub Copilot
- Cursor

理由：

- 用户心智最强
- 参考项目已有成熟实现
- TokenFlow 的价值最容易被用户立刻理解

### P1：很值得加，但不必拖慢首版
- Kiro
- Warp
- GLM / z.ai
- Trae

理由：

- 额度结构清晰
- 技术路径较明确
- 能拉开与只支持大厂平台的差异

### P2：可以先做账号识别或占位
- Gemini
- OpenCode
- Antigravity
- Windsurf / Factory
- JetBrains AI
- OpenRouter

理由：

- 有价值，但部分平台公开额度路径不稳定
- 更适合在适配器框架稳定后扩展

## 8. 对 TokenFlow 产品范围的建议
## 8.1 初版建议保留的核心
- 多平台
- 多账号
- 标准 OAuth / 浏览器跳转
- 统一额度面板
- 明确重置时间
- 本地持久化登录态

## 8.2 初版建议不要立刻做的内容
- 本地代理服务
- 模型路由与 fallback
- 自动配置所有 agent
- token 成本分析
- tunnel / remote mode
- status page 联动
- 太重的企业协作能力

理由很简单：

这些都是高价值，但它们属于“第二层产品能力”，不是 TokenFlow 初版验证价值所必须的东西。

## 9. 最终判断
### 9.1 市场机会
有机会，而且机会真实存在。

因为现在已经出现了两类清晰需求：

- 用户同时订阅多个 AI Coding 平台，不知道哪个还有额度
- 不同平台额度形态不同，用户需要一个统一视图

这正是 TokenFlow 的核心价值。

### 9.2 最现实的产品路径
TokenFlow 最好的路径不是一上来做成 `Quotio + CLIProxyAPI + CodexBar` 的全集成产品，而是：

1. 先做统一 quota monitor
2. 再做多源 provider adapter
3. 再决定要不要加代理、路由、agent 配置

### 9.3 一句话建议
TokenFlow 初版应该把自己定位成：

**一个面向开发者的多平台 AI Coding 订阅额度监控器，而不是一个全功能代理中台。**
